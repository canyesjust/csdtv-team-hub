# Hub UX Quality Control — 2026-07-15

Scope: a clean, consistent, broadcast-fast pass across the hub. This is a
code-and-design-system review, not a pixel screenshot pass. I can't render your
running app from here. If you run it locally I can do a live visual pass over the
real screens and catch anything pixel-level.

## Verdict

Good bones, inconsistent skin. You built a real design-token system in
`app/globals.css` — dark and light themes, a type scale, spacing rhythm, focus
rings, status colors, and a Canyons-yellow accent. The problem is adoption. Most
pages ignore the tokens and hardcode their own colors, so the product drifts
screen to screen. Fixing that, plus the theme default and a couple of speed
gaps, is what moves this from "works" to "clean and fast on air."

Nothing here is a rewrite. It's mostly wiring existing pages to the system you
already have.

## P1 — Consistency and trust

### 1. Color drift: the design system exists but pages bypass it
`app/globals.css` defines the right tokens: `--brand-primary #5ba3e0`,
`--brand-accent #fbae42` (Canyons yellow), `--status-danger #ef4444`,
`--status-success #22c55e`, `--text-primary`, surface layers, and more. But:

- **10 files define their own inline `const C = {}` palette**, and they don't
  match each other or the tokens. Examples:
  - `app/control/[productionId]/program/ProgramClient.tsx` — accent `#4f9dee`,
    live `#ff5d5d`, yea `#34d399`
  - `.../board-meetings/[productionId]/control/ConsoleView.tsx` — bg `#0a0f1a`,
    accent `#4f9dee`, live `#ff5d5d`, amber `#fbbf24`
  - `app/board/components/BoardDaisView.tsx` — bg `#102441`, accent `#f5b53f`
    (a different yellow), blue `#4f9fe0`
  - `app/layout/page.tsx` — a whole separate light palette (`#3C5A99` accent)
- **30+ files hardcode 20 to 70 raw hex values each.** Top offenders:
  `app/control/control-surface.css` (71), `app/dashboard/productions/[id]/page.tsx`
  (70), `app/signage/screen/[code]/signage-screen.css` (58), `ConsoleView.tsx`
  (58), `app/board/components/BoardLiveView.tsx` (57), `app/dashboard/equipment/page.tsx`
  (50).
- **Only a dozen or so files actually use `var(--…)` tokens** (e.g.
  `dashboard/settings`, `dashboard/tasks`, `dashboard/schedule`).

The tell is near-misses. Your "live red" is `#ff5d5d` in the control views but
`--status-danger` is `#ef4444`. Your accent blue is `#4f9dee` in control vs
`--brand-primary #5ba3e0` in tokens. Your yellow is `#f5b53f` or `#fbbf24` vs
`--brand-accent #fbae42`. Same intent, slightly different values, which is
exactly what makes a product feel unpolished and makes a future re-theme painful.

**Why it matters on air:** operators move between the dashboard, control, and
board surfaces during a show. When each has its own reds and blues, "red = live /
danger" stops being a reliable signal. Consistent color is how someone reads a
screen in half a second.

**Fix:** adopt the tokens in **operator-facing chrome only**. Replace inline
`const C` with CSS variables on the dashboard pages and operator consoles (e.g.
`ConsoleView.tsx`, the dashboard list/detail pages). Then add a lint rule that
flags raw hex in those files so drift can't creep back.

**Do NOT migrate the on-air output surfaces.** Verified: the board channel
overlay (`app/board/[channel_number]/overlay/`) uses `.broadcast-overlay-root`,
which `globals.css` intentionally excludes from the theme, and the board
dais/stream views are on-camera graphics. Their fixed palettes are deliberate and
must not follow an operator's dark/light setting. If you want to tame those too,
put them behind a separate **brand-locked** constant, not the themeable UI tokens.

**Safety verdict:** needed for polish, not a bug. Won't break functionality
(colors only). Cosmetic-only risk, so verify each surface visually after
converting. Low risk if you respect the on-air exclusion above.

### 2. Theme is locked to light, then flips — and control surfaces are always dark
`app/layout.tsx` hardcodes `<html className="light">`. `lib/theme.tsx` then
swaps the class to dark on mount from `localStorage`. So a dark-mode user gets a
flash of light on every load (FOUC). Meanwhile the control, board, and program
surfaces are hardcoded dark regardless of theme. The result: a light dashboard
shell wrapping dark broadcast screens, plus a visible flicker.

**Why it matters on air:** control rooms are dark, and a white dashboard flashing
up is both jarring and a reflection risk near cameras.

**Fix, split into two:**

- **FOUC fix (needed, safe).** Set the theme before first paint with a tiny
  inline script in `<head>` that reads the stored preference and sets the class.
  Add `suppressHydrationWarning` to the `<html>` tag so React doesn't complain
  about the class the script set. This removes the flash for dark users and
  can't break anything.
- **Default to dark (optional, verify first).** This is a product choice, not a
  bug fix. Verified: dark mode is already a shipped feature (toggle in
  `settings`, and pages branch on `theme` — `productions/[id]` and `equipment`
  handle both modes), so dark generally works. But changing the default flips
  every user, so spot-check the heavier hardcoded pages in dark before you do it.

**Safety verdict:** FOUC fix is safe and worth doing now. Default-dark is safe to
try but verify first; it's reversible.

## P2 — Speed during a broadcast

### 3. Almost no loading states
Zero `loading.tsx` files and only two components with any skeleton or
`aria-busy`. Pages that fetch (and several poll every 2s) show blank or jump when
data lands.

**Why it matters:** during a show a blank panel reads as "broken," and the
operator hesitates. A skeleton reads as "coming," and they wait a beat.

**Fix:** add `loading.tsx` to the high-traffic live routes (control, board,
productions, tasks) and an `aria-busy` state on the polled panels. Small effort,
big perceived-speed gain.

### 4. Navigation is broad, so live actions are buried
`lib/dashboard-nav.ts` defines many top-level items across role variants, and
there's no obvious one-click path to "go to the live control surface."

**Correction from double-check:** the two "Brand library" entries are NOT a bug.
They're role-gated — `BRAND_LIBRARY_MANAGE` (`/dashboard/brand`) for managers,
`BRAND_LIBRARY_PUBLIC` (`/brand`) for everyone else — and the builder picks
exactly one per user. Leave them alone. Deduping would break the role split.

**Why it matters:** the thing you reach for most under time pressure should be
the easiest to hit, not one of twenty peers in a list.

**Fix:** add a persistent, high-contrast "Control / Go Live" action in the shell
that jumps straight to the active production's control surface. You already have a
`SearchPanel` in the shell — lean on it as a command palette (keyboard-first jump
to any production or action), which is the fastest navigation for a power user.

**Safety verdict:** additive. A new shortcut and a command palette don't touch
existing routes. Safe.

### 5. Confirm live buttons are big and reachable
The control surfaces use `vw`-based sizing (good for a fixed broadcast display),
but verify the primary live actions are large touch/click targets with clear
on/off state. Under pressure, small or ambiguous buttons cause misfires.

**Fix:** during the live visual pass, size-check the primary control buttons and
make active/armed states unmistakable (fill, not just a border).

## P3 — Polish that compounds

### 6. Consolidate shared UI
You already have `lib/ui/styles` (`statusBadge`, `uiStyles`, `statusTone`) and a
clean inline icon set in `AppLayout.tsx`. Good. But buttons, cards, and inputs
are re-styled per page. Pull the common ones into token-based shared components
so a fix lands everywhere at once and new pages start consistent.

### 7. Accessibility
Focus rings are defined globally in `globals.css` — nice. Watch contrast on the
muted grays (`--text-muted`, and the various `soft`/`dim` inline values) over the
darker surfaces. Verify the muted text on `--surface-1` clears WCAG AA. This is
easy to check once and fix at the token level.

### 8. Responsive / operator tablets
No explicit `viewport` export was found; Next provides a default. If any operator
uses a tablet or phone for control, confirm the shell and control surfaces hold
up at those widths. The mobile nav already has a "More" overflow, which is a good
sign it was considered.
